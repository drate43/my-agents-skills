# Prompt Rewrite

사용자의 원본 프롬프트를 받아서 최적화된 버전으로 변환하고, 사용자 승인 후 실행하는 스킬이다.

## 실행 흐름

### 1단계: 원본 분석

핵심 의도, 암묵적 컨텍스트, 모호한 지점, 빠진 제약 조건을 파악한다.

### 2단계: 리라이트

| 원칙 | 핵심 |
|------|------|
| **구체화** | 모호한 표현 → 구체적 지시. "잘" → 무엇이 잘인지 정의 |
| **구조화** | 복합 요청은 단계 분해. 단순 요청은 단순하게 유지 |
| **출력 포맷** | 코드 → 언어 명시, 분석 → 표/불릿, 비교 → 표. 간단한 답변엔 강제 안 함 |
| **토큰 효율** | 중복 제거, 공손 표현 삭제, 핵심 앞배치 |
| **의도 보강** | 사용자 의도 범위 내에서 빠진 요소 보충. 과도한 추가 금지 |
| **역할 프레임** | 전문성 필요 시에만 추가. 단순 질문엔 불필요 |

### 3단계: 사용자에게 제시

아래 포맷으로 원본과 최적화 버전을 나란히 보여준다:

```
## 원본 프롬프트
> {사용자의 원본 프롬프트}

## 최적화 프롬프트

{리라이트된 프롬프트}

---
**변경 요약**: {무엇을 왜 바꿨는지 1-3줄}

이대로 실행할까요? (Y/수정 사항 알려주세요)
```

### 4단계: 승인 및 실행

- 사용자가 "Y", "ㅇ", "응", "좋아", "실행" 등 긍정 응답 → 최적화된 프롬프트를 그대로 실행
- 사용자가 수정 요청 → 반영 후 다시 제시
- 사용자가 "원본대로" → 원본 프롬프트 그대로 실행

## 명확도 판정: 리라이트 여부 결정

1단계 분석 후, 리라이트가 실제로 가치를 더하는지 먼저 판정한다. 모든 프롬프트를 무조건 리라이트하는 것은 오히려 오버헤드다.

**패스스루 (리라이트 없이 원본 그대로 실행)**:
- 한 단어 명령어 (예: "커밋", "빌드", "테스트")
- 슬래시 커맨드 호출 (예: "/commit", "/review-pr")
- 이미 구체적 지시 + 출력 포맷 + 제약 조건이 갖춰진 프롬프트

→ "이미 충분히 명확한 프롬프트입니다. 그대로 실행합니다." 라고 알려준다.

**경량 리라이트 (원본 골격 유지, 최소 보강만)**:
- 의도가 명확하지만 출력 포맷만 빠진 경우 → 포맷만 추가
- 코드가 포함된 디버깅 요청 → 코드 블록 정리 + 점검 포인트 1-2개 추가
- 단일 작업 요청 ("메일 써줘", "번역해줘") → 톤/길이/포맷 보강만

핵심 원칙: **원본이 10단어면 최적화도 비슷한 밀도를 유지한다.** 짧은 요청을 장문 프롬프트로 부풀리지 않는다. 추가하는 내용은 실제로 결과 품질에 영향을 주는 것만.

**풀 리라이트 (구조 재설계)**:
- 여러 작업이 뒤섞인 복합 요청
- 모호한 범위 ("정리해줘", "개선해줘", "만들어줘")
- 약어/줄임말이 많아 해석이 필요한 경우
- 전문 도메인 지식이 필요한 요청

## 리라이트 강도 예시

| 원본 | 판정 | 결과 |
|---|---|---|
| "커밋" | 패스스루 | 그대로 실행 |
| "이 코드 왜 안 돼? `fetch('/api')`" | 경량 | 코드블록 정리 + 점검 포인트 2-3개 추가 (5줄 이내) |
| "메일 써줘 다음주 화요일에" | 경량 | 날짜 변환 + 톤/길이 명시 (원본 길이의 2-3배 이내) |
| "결제 플로우 정리해줘" | 풀 | 범위/항목/출력포맷 전체 구조화 |
| "번장에 등록하고 매일 천원씩 내려" | 풀 | 약어 해석 + 단계 분해 + 실현 가능성 조건 추가 |

## 주의사항

- **의도 변경 금지**: 리라이트는 의도를 더 명확하게 만드는 것이지, 바꾸는 것이 아니다
- **비례 원칙**: 원본의 복잡도에 비례해서 리라이트한다. 간단한 요청에 10단계 프로세스를 만들지 않는다
- **언어 유지**: 사용자가 한국어로 썼으면 한국어로 리라이트
- **톤 유지**: 캐주얼한 요청을 논문 스타일로 바꾸지 않는다 — 구조만 개선
- **부풀리기 금지**: 리라이트가 원본보다 5배 이상 길어지면 무언가 잘못된 것이다. 다시 줄여라
