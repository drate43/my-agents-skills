# Review PRD — PRD 공정 리뷰 스킬

PRD 작성 컨텍스트를 **의도적으로 배제**하고, 독립 에이전트가 문서만 보고 리뷰한다.

## 왜 컨텍스트 분리가 필요한가

같은 컨텍스트에서 만들고 리뷰하면:
- **확증 편향**: 자기가 쓴 논리를 자기가 옳다고 판단
- **맥락 보완**: 대화에서 논의했지만 문서에 빠진 내용을 "당연히 있다"고 착각
- **범위 고착**: 작성 중 합의한 범위를 의심 없이 수용

독립 에이전트는 **문서에 적힌 것만** 보고 판단하므로, 빠진 내용과 모호한 표현을 정확히 잡아낸다.

## 워크플로우

### STEP 1: PRD 파일 찾기

- 인자가 있으면 해당 파일 사용 (예: `/review-prd docs/PRD-checkout-refactor.md`)
- 인자가 없으면 `docs/PRD-*.md` 검색 → 여러 개면 사용자에게 선택 요청
- PRD가 없으면 → `write-prd` 먼저 사용하도록 안내

### STEP 2: 독립 에이전트 Spawn

**반드시 Agent 도구를 사용**하여 서브에이전트를 spawn한다.

에이전트에게 전달할 프롬프트 템플릿:

```
너는 PRD 리뷰어다. 아래 PRD 파일을 읽고 코드베이스를 독립적으로 탐색하여 공정하게 리뷰해라.

## 리뷰 대상
PRD 파일: {prd_file_path}

## 리뷰 체크리스트

### 1. 완전성 (Completeness)
- [ ] 배경/문제 정의가 충분한가
- [ ] 사용자 스토리가 모든 주요 시나리오를 커버하는가
- [ ] 기능 요구사항이 사용자 스토리와 1:1 대응하는가
- [ ] 엣지 케이스가 충분한가 (네트워크 실패, 빈 데이터, 동시 요청, 플랫폼 차이)
- [ ] 범위 외가 명시적이고 이유가 타당한가
- [ ] 성공 기준이 측정 가능한가

### 2. 정확성 (Accuracy)
- [ ] 영향받는 파일의 `파일경로:라인번호`가 실제 코드와 일치하는가 (Read 도구로 검증)
- [ ] 기존 코드 동작 설명이 실제 구현과 일치하는가
- [ ] API 엔드포인트가 실제로 존재하는가
- [ ] "dead code" 판단이 정확한가 (실제 도달 불가능한지 흐름 추적)

### 3. 일관성 (Consistency)
- [ ] 프로젝트의 기존 아키텍처 패턴과 일관적인가
- [ ] 유사 도메인의 기존 구현과 패턴이 일치하는가
- [ ] 이슈 분해 간 선행 조건이 논리적인가

### 4. 리스크 & 누락 (Risks & Gaps)
- [ ] 빠진 엣지 케이스는 없는가
- [ ] 범위 외로 밀린 항목 중 MVP에 포함해야 할 것은 없는가
- [ ] 보안 고려사항 (토큰, 인증, XSS 등)이 누락되지 않았는가
- [ ] 플랫폼별 차이 (iOS/AOS/웹)가 충분히 고려되었는가
```

### STEP 3: 결과 전달

에이전트 결과를 사용자에게 전달하며, 다음 액션을 안내:

```
## 다음 단계

- 문제 발견 시 → PRD 수정 후 `/review-prd` 재실행
- 리뷰 통과 시 → `/prd-to-issues`로 이슈 분해 진행
```

## 규칙

1. **컨텍스트 분리 필수** — 반드시 Agent 도구로 서브에이전트를 spawn. 메인 컨텍스트에서 직접 리뷰하지 않음
2. **코드 검증 필수** — PRD의 `파일경로:라인번호` 주장을 Read 도구로 실제 확인
3. **문서에 적힌 것만 판단** — 대화 맥락이나 암묵적 합의에 의존하지 않음
4. **수정은 하지 않음** — 리뷰 결과만 제시, PRD 파일 수정은 사용자가 판단
5. **한국어로 응답**
