# Codebase Health — 코드베이스 구조 진단 스킬

코드베이스의 구조적 건강도를 진단하고 **AI 친화적 개선 후보**를 제시한다.

## 진단 관점

### 1. Shallow vs Deep Module 분석

| 진단 | 설명 | 증상 |
|------|------|------|
| **Shallow** | 파일이 너무 잘게 쪼개져 있어 하나의 개념을 이해하려면 여러 파일을 탐색 | 한 기능 수정에 5개 이상 파일 변경 필요 |
| **Deep** | 얇은 인터페이스 뒤에 풍부한 기능이 숨어있음 | 호출 1곳 수정으로 동작 변경 가능 |

→ Shallow 모듈을 Deep 모듈로 통합할 후보 제시

### 2. 테스트 가능성 vs 통합 버그

- 순수 함수 추출이 **실제 통합 버그를 숨기는 패턴** 식별
- 단위 테스트는 통과하지만 통합 시 실패하는 지점 탐지

### 3. 커플링 위험도

- 모듈 간 밀결합 식별 (순환 import, 공유 상태, 암묵적 의존)
- 변경 시 연쇄 영향이 큰 핫스팟 파일 식별

## 출력 형식

```
## Codebase Health Report

### 요약
- 진단 대상: [전체 / 특정 영역]
- 진단 일시: YYYY-MM-DD

### Shallow Module 후보 (Deep 통합 권장)
| 현재 구조 | 파일 수 | 통합 제안 | 효과 |
|---------|--------|---------|------|
| 설명 | N개 | 제안 | 효과 |

### 통합 버그 위험 지점
| 위치 | 위험 | 근거 |
|------|------|------|
| `파일경로:라인` | 설명 | 근거 |

### 커플링 핫스팟
| 파일 | 의존하는 곳 | 변경 영향 범위 |
|------|-----------|-------------|
| `경로` | N곳 | 설명 |

### 권장 액션 (우선순위순)
1. [높음] 설명 → **다음 단계**: `propose-fix`로 구현 방안 요청
2. [중간] 설명 → **다음 단계**: `propose-fix` 또는 직접 수정
3. [낮음] 설명 → 참고용, 즉시 조치 불필요
```

## 다음 단계 연계

진단 결과에서 개선이 필요한 항목이 나오면:
- **구체적 구현 코드가 필요** → `propose-fix` 스킬 호출 안내
- **구조 변경 전 논의 필요** → `grill-me` 스킬로 설계 분기 탐색 안내
- **여러 파일에 걸친 대규모 리팩토링** → `dev-cycle` 에이전트 추천

## 규칙

1. **코드를 직접 읽고 판단** — 추측 금지
2. **`파일경로:라인번호` 형식** 필수
3. 개선 제안은 **구체적** (어떤 파일을 어떻게 통합할지)
4. 현재 동작을 깨뜨리는 제안 금지 — 안전한 리팩토링만
5. 진단만 하고 끝내지 않음 — **다음 단계를 반드시 안내**
