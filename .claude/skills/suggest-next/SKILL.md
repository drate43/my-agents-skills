# Suggest Next — 파이프라인 내비게이터

현재 상황을 파악하고 **지금 해야 할 단계**를 안내한다.

## 파이프라인

```
① grill-me → ② write-prd → ③ prd-to-issues → ④ investigate-issue / dev-cycle
   (심문)       (PRD 작성)     (이슈 분해)         (조사 또는 구현)
                                                      ├─ plan-first
                                                      ├─ tdd
                                                      └─ pre-commit-review
```

## 판단 로직

### 파이프라인 단계 (큰 기능 개발)

| 현재 상황 | 추천 단계 | 판단 기준 |
|---------|---------|---------|
| 아이디어만 있고 구체화 안 됨 | ① `grill-me` | 사용자가 "~하려고 하는데", "어떻게 하면 좋을까" 등 탐색적 질문 |
| 주요 설계 분기가 모두 해소됨 | ② `write-prd` | 이 대화에서 5개 이상 Q&A가 오갔거나, 사용자가 "정리해줘" 요청 |
| PRD 있고, 이슈 분해 안 됨 | ③ `prd-to-issues` | `docs/PRD-*.md` 파일이 존재하고 이슈 분해가 안 됨 |
| 이슈 있고, 조사/분석 필요 | `investigate-issue` | 버그, 장애, 원인 분석이 필요한 이슈 |
| 이슈 분해 완료, 구현 시작 | ④ `dev-cycle` | 구현 대상이 명확하고 코드 작성이 필요 |

### 작은 변경 (파이프라인 생략)

**작은 변경 기준**: 아래 **모두** 해당하면 파이프라인 없이 바로 진행:
- 변경 파일 3개 이하 (추정)
- 기존 패턴을 따르는 단순 수정 (새 아키텍처 도입 없음)
- 사용자가 이미 무엇을 해야 하는지 알고 있음

→ `plan-first` 에이전트로 바로 진행하거나, 1줄 수정이면 직접 구현

### 유틸리티 (언제든)

| 상황 | 추천 | 판단 기준 |
|------|------|---------|
| 커밋/푸시/PR 요청 | `pre-commit-review` | "커밋해줘", "푸시해줘", "PR 만들어줘" |
| 코드 위치/흐름 질문 | `find-in-code` | "어디있어?", "누가 호출해?", "흐름 알려줘" |
| 이미 존재하는 두 코드 비교 | `diff-review` | "뭐가 달라졌어?", "diff 보여줘" |
| 구현 방안이 필요 (코드 언급 있음) | `propose-fix` | "이거 고치려면?", "어떻게 할까?" + 특정 파일/함수 언급 |
| 코드 구조 점검 | `codebase-health` | "구조 점검", "리팩토링 후보" → 결과 후 `propose-fix`로 이어갈 수 있음 |

## 출력 형식

```
## 현재 위치: ② write-prd

✅ ① grill-me — 완료
👉 ② write-prd — 지금 이 단계. `/write-prd` 호출하세요
⬜ ③ prd-to-issues — 다음
⬜ ④ investigate-issue / dev-cycle — 대기

💡 단독 실행: plan-first, tdd, pre-commit-review
🔧 유틸리티: find-in-code, diff-review, propose-fix, codebase-health
```

## 규칙

1. **현재 대화 맥락에서 판단** — 이미 5개 이상 Q&A가 오갔으면 grill-me 완료로 간주
2. **추천이지 강제가 아님** — "이 단계를 추천합니다"로 표현
3. **작은 변경은 파이프라인 생략** — 위 기준 참고
4. **구체적 코드 언급이 있으면 propose-fix** — "기능 추가해줘"만으로는 suggest-next, "이 파일의 이 함수 고쳐줘"면 propose-fix
5. 한국어로 응답
