여러 모듈/파일에 동일 패턴이 적용되어야 할 때, 누락 없이 전부 적용됐는지 매트릭스 표로 검증하는 스킬.

## 언제 사용하는가

- 사용자가 "빠진 거 없어?", "전부 적용됐어?", "누락 체크" 등 일관성 확인을 요청할 때
- 여러 모듈에 동일 패턴(로그, 에러 핸들링, 훅 호출 등)을 추가/수정한 직후 검증
- 코드 분석 중 "이 패턴이 모든 관련 파일에 있는가?"를 확인해야 할 때
- 플랫폼별, 페이지별, 컴포넌트별 등 동일 기능의 구현 일관성을 점검할 때

## 실행 절차

### 1단계: 대상 식별

체크할 **패턴**(행)과 **모듈**(열)을 파악한다.

- **패턴**: 적용되어야 하는 항목 (예: 로그 이벤트, 에러 핸들링, 특정 훅 호출)
- **모듈**: 패턴이 적용되어야 하는 파일/컴포넌트 단위

### 2단계: Grep/Read로 실제 코드 검증

각 모듈에서 각 패턴의 존재 여부를 Grep 또는 Read로 **실제 확인**한다. 추측하지 않는다.

### 3단계: 매트릭스 표 출력

```markdown
| 패턴 | 모듈A | 모듈B | 모듈C | 모듈D |
|------|:-----:|:-----:|:-----:|:-----:|
| 패턴1 | ✓ | ✓ | **✗** | ✓ |
| 패턴2 | ✓ | **✗** | ✓ | N/A |
| 패턴3 | ✓ | ✓ | ✓ | ✓ |
```

### 4단계: 누락 요약

표 아래에 누락 항목만 모아서 정리:

```markdown
### 누락 항목
- **모듈B** — `패턴2` 없음
- **모듈C** — `패턴1` 없음
```

## 출력 규칙

### 셀 값 — 심볼만, 부가 정보 금지
| 심볼 | 의미 |
|------|------|
| ✓ | 해당 패턴이 모듈에 존재 |
| **✗** | 누락 — **굵게** 표시하여 시각적 강조 |
| N/A | 해당 모듈에는 적용 불필요 (예: 데스크톱 전용 패턴 vs 모바일 모듈) |

셀 안에는 ✓ / **✗** / N/A **만** 넣는다. 라인 번호, 파일 경로, 설명 등 부가 정보는 셀에 절대 넣지 않는다. 표가 깔끔해야 한눈에 파악 가능하기 때문이다.

### 열 헤더 — 공백 구분, 슬래시 금지
- `module-a desktop` (O) / `module-a/desktop` (X)
- 중복 파일명이면 구분 가능한 최소 경로를 공백으로 연결 (예: `module-a Process.desktop`)

### 코드 위치 — 누락 요약에서만 제공
- 누락 항목 요약에서 "어디에 추가해야 하는지" 힌트로 파일:라인 제공
- 매트릭스 표 본체에는 코드 위치를 넣지 않는다

## 주의사항

- N/A 판단은 보수적으로: 확실히 불필요한 경우에만 N/A, 애매하면 ✗로 표시하고 주석 추가
- 모듈이 5개 이상이면 표가 길어지므로, 관련성 기준으로 그룹핑 고려
- 패턴 존재 여부는 반드시 Grep/Read로 실제 확인 후 표시 (추측 금지)
