# Frontend Design Guide

**좋은 프론트엔드 코드 = 변경하기 쉬운 코드.**

4가지 설계 원칙의 규칙 요약과 체크리스트. Bad/Good 코드 예시가 필요하면 `references/examples.md`를 참조한다.

---

## 1. 가독성 (Readability)

한 번에 고려하는 맥락이 적고, 위에서 아래로 자연스럽게 읽히는 코드.

> 하위 원칙: **맥락 줄이기** + **이름 붙이기** + **위에서 아래로 읽히게 하기**

| # | 규칙 | 핵심 |
|---|------|------|
| 1-1 | 같이 실행되지 않는 코드 분리 | 동시에 실행되지 않는 분기를 별개 컴포넌트로 추출 |
| 1-2 | 구현 상세 추상화 | 복잡한 로직을 Wrapper/Hook으로 캡슐화하여 인지 부하 감소 |
| 1-3 | 로직 종류별 함수 쪼개기 | 쿼리 파라미터·상태·API 등 다른 종류의 로직을 하나의 Hook에 합치지 않기. 불필요한 리렌더링 방지에도 기여 (응집성·결합도 관점에서도 동일 원칙) |
| 1-4 | 복잡한 조건에 이름 붙이기 | 불린 조건을 `isSameCategory` 같은 의미 있는 변수명에 할당 |
| 1-5 | 매직 넘버에 이름 붙이기 | 하드코딩 숫자/문자열을 상수로 추출. 응집도에도 영향 — 관련 값이 함께 수정되어야 하므로 |
| 1-6 | 시점 이동 줄이기 | 코드 읽을 때 여러 파일/함수를 넘나드는 횟수 최소화. 선택지가 적으면 인라인 객체 고려 |
| 1-7 | 삼항 연산자 단순하게 하기 | 중첩 삼항은 IIFE + if/else로 대체 |
| 1-8 | Early Return으로 중첩 제거 | 중첩 if를 가드 절로 평탄화 |
| 1-9 | 컴포넌트 내부 선언 순서 통일 | hooks → derived state → handlers → effects → early returns → render |
| 1-10 | JSX 인라인 함수 추출 | 3줄 이상 인라인 함수는 별도 핸들러로 추출 |
| 1-11 | Boolean Props 긍정형 | 이중 부정 방지. `isHidden` 대신 `visible`, 조건문에서 `!isNotX` 같은 패턴 제거 |

## 2. 예측 가능성 (Predictability)

이름, 파라미터, 반환값만으로 동작을 예측할 수 있는 코드.

| # | 규칙 | 핵심 |
|---|------|------|
| 2-1 | 고유하고 설명적인 이름 | 라이브러리와 래퍼가 같은 이름이면 혼란. `http` → `httpService.getWithAuth`. 역할이 다르면 이름도 달라야 한다 (`logEvent` → `sendAnalyticsEvent`) |
| 2-2 | 반환 타입 통일 | 같은 종류 함수/훅은 같은 형태를 반환. `useUser()`와 `useServerTime()`이 각각 query 객체와 data를 반환하면 혼란 |
| 2-3 | 숨은 로직 드러내기 | 함수 시그니처에 없는 부작용(로깅, 리다이렉트)은 호출부에서 명시. 숨은 로깅이 장애를 유발할 수 있음 |
| 2-4 | 파라미터 2개 초과 시 객체 전환 | 인자 의미를 명시적으로. 호출부만 보고 각 인자 역할을 알 수 있어야 함 |
| 2-5 | 불가능한 상태 제거 | Discriminated Union으로 타입 레벨에서 불가능한 조합 차단. `status: 'loading'`인데 `data`가 존재하는 상태를 원천 차단 |
| 2-6 | useEffect 의도 명시 | 복잡한 조건부 useEffect는 `useOnceWhen` 같은 커스텀 훅으로 의도 표현 |
| 2-7 | 핸들러 네이밍 일관성 | Props 콜백 = `onXxx`, 내부 핸들러 = `handleXxx`. 혼재하면 데이터 흐름 추적이 어려움 |

## 3. 응집성 (Cohesion)

수정되어야 할 코드가 항상 같이 수정되는 정도.

> **응집도 vs 가독성 트레이드오프**: 위험성이 높으면 응집도 우선, 그렇지 않으면 가독성 우선.

| # | 규칙 | 핵심 |
|---|------|------|
| 3-1 | Co-location | 함께 변경되는 파일을 같은 디렉토리에. 모듈 종류별(X) → 도메인 기반(O). 기능 삭제 시 디렉토리 통째로 제거 가능해야 함 |
| 3-2 | 폼 응집도 선택 | 변경 단위에 따라 필드 수준(독립 검증, 재사용) or 폼 전체 수준(Zod, 필드 간 의존성) |
| 3-3 | Feature Hook | 컴포넌트에 흩어진 도메인 로직을 하나의 훅에 응집. 컴포넌트는 UI만 담당 |
| 3-4 | 컴포넌트 크기 제한 | 한 화면에서 전체를 파악할 수 없으면 분리 신호. 경험적으로 150~200줄 초과 시 역할별 하위 컴포넌트 추출을 검토 (절대 기준이 아닌 점검 트리거) |
| 3-5 | API 변환 응집 | API 응답 변환은 API 레이어에서 완료. 컴포넌트는 정제된 데이터만 사용. 응답 형태 변경 시 API 파일만 수정하면 됨 |

## 4. 결합도 (Coupling)

코드 수정 시 영향범위. 영향범위가 적고 예측 가능한 코드가 수정하기 쉽다.

| # | 규칙 | 핵심 |
|---|------|------|
| 4-1 | 중복 코드 허용 | "변경의 독립성 > 중복 제거". 사례가 분기될 수 있으면 조기 추상화 금지. 정확히 같고 앞으로도 안 변할 때만 공통화 |
| 4-2 | Props Drilling 제거 | children 합성(조합 패턴) 또는 Context API로 중간 단계 전달 제거. props 10개짜리 모달도 children 합성으로 껍데기만 남기기 |
| 4-3 | 외부 라이브러리 래핑 | Adapter 패턴으로 교체 비용을 1곳으로 제한. 30곳에서 직접 import하면 교체 시 30곳 수정 |
| 4-4 | 형제 간 공유 상태 추출 | 부모 중계 대신 전용 store/훅으로 직접 구독. 부모 컴포넌트의 상태 과다 방지 |
| 4-5 | API 의존성 역전 | 훅이 URL/응답 구조를 모르게. API 함수를 통해 간접 접근. 엔드포인트 변경 시 API 파일만 수정 |
| 4-6 | Headless 패턴 | 복잡한 UI는 로직(훅)과 렌더링(컴포넌트) 분리. 같은 로직으로 다른 UI를 만들 수 있게 |

## 리뷰 체크리스트 (Quick Reference)

### 가독성
- [ ] 같이 실행되지 않는 코드가 한 컴포넌트에 있는가? → 분리
- [ ] 구현 상세가 노출되어 있는가? → Wrapper/Hook 추상화
- [ ] 여러 종류 로직이 하나의 Hook에 있는가? → 개별 Hook 분리
- [ ] 매직 넘버/스트링이 있는가? → 상수 추출
- [ ] 중첩 삼항 또는 3단계+ 중첩 if? → IIFE/Early Return
- [ ] 시점 이동 3회 이상? → 인라인화
- [ ] JSX 내 3줄+ 인라인 함수? → 핸들러 추출

### 예측 가능성
- [ ] 이름이 겹치거나 모호한가? → 고유하고 설명적인 이름
- [ ] 같은 종류 함수가 다른 반환 타입? → 통일
- [ ] 숨겨진 부작용? → 호출부 이동
- [ ] 파라미터 3개+? → 객체 전환
- [ ] 불가능한 상태 조합 허용? → Discriminated Union
- [ ] useEffect 조건 복잡? → 커스텀 훅
- [ ] on/handle 혼재? → Props=on, 내부=handle

### 응집성
- [ ] 함께 변경되는 파일이 다른 폴더? → Co-location
- [ ] 도메인 로직이 컴포넌트에 흩어짐? → Feature Hook
- [ ] 컴포넌트가 한 화면에 안 들어옴? → 하위 컴포넌트 추출
- [ ] API 변환이 컴포넌트에? → API 레이어 이동
- [ ] 응집도 vs 가독성 상충? → 위험도 높으면 응집도 우선

### 결합도
- [ ] 성급하게 공통화했는가? → 변경 독립성 > 중복 제거
- [ ] Props 5개+/3단계+ 전달? → children 합성 or Context
- [ ] 외부 라이브러리 10곳+ 직접 import? → 래퍼
- [ ] 훅이 API URL/응답에 직접 접근? → API 레이어 분리
- [ ] UI 로직+스타일 한 파일? → Headless 패턴

## 응답 가이드

이 스킬을 참조하여 코드 설계 상담에 답할 때 아래 구조를 따른다.

### 응답 구조

1. **문제 진단** — 위반 규칙을 번호(`[1-3]`, `[4-2]` 등)로 나열. 3개 이내로 핵심만.
2. **수정 우선순위** — 여러 문제가 있으면 운영안정성 > 결합도 > 응집성 > 가독성 순서로 번호 매기기. 각 항목에 "왜 이것이 먼저인지" 한 줄 근거.
3. **해결 방안** — 각 문제에 대해 Bad/Good 코드 쌍. 방안이 2개 이상이면 상황별 선택 기준 표.
4. **과도 적용 주의** — "이 경우에는 하지 마세요" 1~2개. 점진적 적용 전략 포함.

### 톤

- 간결하게. 설명 1줄 + 코드로 보여주기. 장황한 서론 금지.
- "~할 수 있습니다" 대신 "~하세요" 직접 지시형.
- 코드 예시는 프로젝트 맥락에 맞게.

> Bad/Good 코드 예시가 필요하면 `references/examples.md`를 읽어서 해당 규칙 번호의 예시를 참조한다.
