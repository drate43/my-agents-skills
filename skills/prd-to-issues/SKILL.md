# PRD to Issues — PRD 이슈 분해 스킬

PRD가 **목적지**라면, 이 스킬은 그곳까지의 **여정**을 설계한다.
PRD를 읽고 코드베이스를 탐색하여, 독립적으로 작업 가능한 vertical slice 이슈 목록을 생성한다.

## 워크플로우

```
[PRD 입력]
    ↓
[STEP 1] PRD 확인 & 로드
    ↓
[STEP 2] 코드베이스 탐색
    ↓
[STEP 3] Vertical Slice 분해
    ↓
[STEP 4] 사용자 리뷰 & 조정
    ↓
[STEP 5] 이슈 등록 (선택)
    ↓
[완료]
```

---

## 각 단계 상세

### STEP 1: PRD 확인 & 로드

PRD 문서를 찾아 로드한다.

- `docs/PRD-*.md` 파일 검색
- 여러 개면 사용자에게 선택 요청
- PRD가 없으면 → `write-prd` 스킬 먼저 사용하도록 안내
- PRD의 **사용자 스토리**와 **기능 요구사항**을 핵심 입력으로 사용

### STEP 2: 코드베이스 탐색

PRD의 기술 요구사항을 기반으로 코드베이스를 탐색한다.

- PRD에 명시된 영향받는 파일/모듈 확인
- 관련 기존 코드 패턴 파악 (재사용 가능한 것, 수정 필요한 것)
- 통합 계층 식별:
  - **UI 계층**: 페이지, 컴포넌트, 스타일
  - **상태 계층**: 훅, 컨텍스트, 스토어
  - **API 계층**: API route, 서비스, 타입
  - **인프라 계층**: 설정, 유틸, 미들웨어
- 발견한 내용을 `파일경로:라인번호` 형식으로 정리

### STEP 3: Vertical Slice 분해

PRD를 **수직 슬라이스**로 분해한다.

> **핵심 원칙**: 각 이슈는 한 계층의 수평 슬라이스가 아닌, **모든 통합 계층을 관통하는 얇은 수직 슬라이스**다.
> Tracer bullet — 미지의 미지(unknown unknowns)를 빠르게 드러낸다.

**수평 슬라이스 (나쁜 예)**:
```
이슈1: 모든 컴포넌트 UI 작성
이슈2: 모든 API 연동
이슈3: 모든 상태 관리
```

**수직 슬라이스 (좋은 예)**:
```
이슈1: 핵심 흐름 A — UI + API + 상태 전체 관통
이슈2: 핵심 흐름 B — UI + API + 상태 전체 관통
이슈3: 엣지 케이스 처리 — 에러 UI + 에러 API + 폴백
```

**분해 기준**:
1. 각 슬라이스는 **단독으로 테스트·배포 가능**해야 함
2. 각 슬라이스는 **사용자 스토리 1개 이상**에 대응
3. 가장 위험한(불확실한) 슬라이스를 **첫 번째**로 배치
4. 슬라이스 간 **blocking 관계**를 명시

**출력 형식**:

```markdown
## 이슈 분해 결과

### 이슈 #{N}: {제목}
- **사용자 스토리**: {대응하는 사용자 스토리}
- **범위**: {이 이슈에서 하는 것}
- **관통 계층**: UI → 상태 → API → (인프라)
- **변경 파일**:
  - `파일경로:라인번호` — 변경 내용 요약
- **선행 조건**: 없음 / 이슈 #X 완료 후
- **병렬 가능**: 이슈 #Y와 병렬 가능
- **예상 복잡도**: S / M / L
```

**요약 테이블**:

| # | 이슈 제목 | 선행 조건 | 병렬 가능 | 복잡도 |
|---|---------|---------|---------|-------|
| 1 | 설명 | 없음 | - | M |
| 2 | 설명 | 없음 | #3과 병렬 | S |
| 3 | 설명 | #1 완료 후 | #2와 병렬 | M |

**실행 순서 다이어그램**:

```
#1 ──→ #3 ──→ #5
       ↗
#2 ──→ #4
```

### STEP 4: 사용자 리뷰 & 조정

분해 결과를 사용자에게 제시하고 피드백을 받는다.

- "이 분해가 맞나요? 수정할 부분이 있으면 알려주세요"
- 사용자 피드백 반영하여 슬라이스 조정
- **사용자 승인 없이 이슈를 생성하지 않는다**

### STEP 5: 이슈 등록 (선택)

사용자가 승인하면 이슈를 등록한다.

- **Jira**: `jira-write` 스킬을 호출하여 티켓 생성
- **GitHub Issues**: `gh issue create` 명령으로 생성
- **문서만**: `docs/` 에 이슈 분해 문서로 저장
- 사용자가 등록 대상을 선택 ("Jira로", "GitHub으로", "문서로만")
- blocking 관계는 이슈 본문에 `Blocked by #N` / `Blocks #N` 형태로 기록

---

## 규칙

1. **vertical slice 필수** — 수평 분해(계층별)는 거부하고, 수직 관통 슬라이스로만 분해
2. **코드베이스 검증 필수** — 변경 파일 목록은 실제 탐색 결과 기반 (`파일경로:라인번호`)
3. **위험도 우선** — 가장 불확실한 슬라이스를 첫 번째 이슈로 배치
4. **사용자 승인 필수** — STEP 4에서 승인 없이 이슈를 생성하지 않음
5. **PRD 없으면 거부** — PRD가 없으면 `write-prd` 먼저 안내
6. **한국어로 응답**
