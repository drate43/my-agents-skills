# Write PRD — 요구사항 문서 작성 스킬

대화에서 도출된 결정사항을 **구조화된 PRD**로 변환한다.

## 워크플로우

1. **입력 확인** — grill-me 완료 여부 판단:
   - **완료 기준**: 주요 설계 분기별 의사결정이 기록됨 (5개 이상 Q&A), 범위(MVP)가 합의됨
   - 완료됐으면 그 결과를 기반으로 작성
   - 완료되지 않았으면 사용자에게 기능 설명 요청 (또는 grill-me 먼저 추천)
2. **코드베이스 탐색** — 사용자 주장을 코드로 검증 (기존 구현, 관련 파일 확인)
3. **보충 질문** — 불명확한 부분이 있으면 grill-me 스타일로 추가 질문
4. **모듈 스케치** — 필요한 주요 모듈/컴포넌트를 대략적으로 파악
5. **PRD 작성** — 아래 템플릿으로 `docs/` 에 저장

## PRD 템플릿

```markdown
# [기능명] PRD

## 배경
- 왜 이 기능이 필요한가
- 해결하려는 문제

## 사용자 스토리
- [ ] AS A [역할], I WANT [행동], SO THAT [가치]
- [ ] AS A [역할], I WANT [행동], SO THAT [가치]

## 기능 요구사항
### 필수 (MVP)
1. 요구사항 설명

### 선택 (추후)
1. 요구사항 설명

## 기술 요구사항
- 영향받는 파일/모듈: `파일경로:라인번호`
- 새로 생성할 파일
- API 변경 사항
- 플랫폼별 고려사항 (iOS/AOS/웹)

## 엣지 케이스
| 시나리오 | 기대 동작 |
|---------|---------|
| 설명 | 동작 |

## 범위 외 (명시적 제외)
- 이번에 하지 않는 것 + 이유

## 성공 기준
- 어떻게 되면 완료인가
```

## 파일명

`docs/PRD-{TICKET-NNN}-{주제-요약}.md` (예: `docs/PRD-PROJ-150-notification-center.md`)
Jira 티켓이 아직 없으면 `docs/PRD-{주제-요약}.md`로 생성 후, 티켓 생성 시 이름 변경

## 규칙

1. **코드베이스 검증 필수** — "~할 것이다"가 아닌 "~가 확인됐다" 근거 기반
2. **사용자 스토리 중심** — 기술 구현이 아닌 사용자 행동 관점으로 기술
3. **범위 외 명시** — 하지 않는 것을 명확히 하여 scope creep 방지
4. **`파일경로:라인번호` 형식** 사용

## 파이프라인 위치

```
① grill-me → 👉 ② write-prd → ③ prd-to-issues → ④ investigate-issue / dev-cycle
```
