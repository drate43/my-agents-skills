# Implementation Proposal 스킬

구현 방식을 제안할 때 아래 구조로 출력한다. 함께 적용해야만 문제가 해결되는 변경들은 **1가지 세트**로 묶어 제시한다.

## 출력 구조

### 1. 근본 원인

| 파일 | 라인 | 문제 |
|------|------|------|
| `절대경로` | :라인 | 설명 |

### 2. 구현 방안 비교 (방안이 2개 이상일 때)

| | A안 | B안 |
|---|---|---|
| **방법** | 설명 | 설명 |
| **수정 범위** | 파일 수 | 파일 수 |
| **기존 코드 영향** | 설명 | 설명 |
| **롤백 용이성** | 높/중/낮 | 높/중/낮 |
| **결론** | 추천/비추천 | 추천/비추천 |

추천 방안과 이유를 한 줄로 명시.

### 3. 사이드이펙트 분석

추천 방안 기준, **영향도별** 정리. 각 항목에:
- 제목 + 영향도 (높/중/낮)
- 시나리오 + 관련 코드 위치
- 위험하면 발생 조건 + 회피 방법

### 4. 최종 구현 코드

파일별 Before/After 코드블록. 변경 이유를 코드 아래 한 줄로.

### 5. 변경 세트 구성

| 순서 | 파일 | 변경 | 역할 |
|------|------|------|------|
| ① | `경로` | 내용 | 역할 |

**누락 시 결과**: ①이 없으면 [문제], ②가 없으면 [문제]
**적용 순서**: 동시 배포 가능 / ① → ② 순서 필수

### 6. 건드리지 않는 것

의도적 제외 항목과 이유를 표로.

### 7. 테스트 방법

**로컬**: 재현 방법 / **QA**: 테스트 케이스 표 / **체크리스트**: 배포 전 확인 항목

## 규칙

1. **절대경로 필수** — `프로젝트루트/src/파일경로:라인번호` 형식
2. 비교표 셀은 짧고 명확하게
3. 인프라 의존성(ALB, CDN, DNS)이 있으면 명시

## 경량 모드

수정 파일 ≤ 2개 AND 방안 1개이면:
- **생략**: 2(비교), 3(사이드이펙트), 6(스코프 외)
- **필수**: 1(원인), 4(코드), 7(테스트)

## 스킬 선택 기준

| 상황 | 사용 스킬 | 판단 기준 |
|------|----------|---------|
| 아직 없는 코드를 새로 제안 | **propose-fix** | "어떻게 할까?", 특정 파일/함수 언급 있음 |
| 이미 존재하는 두 코드 비교 | **diff-review** | "뭐가 달라졌어?", git diff, PR 리뷰 |
| 코드 위치/구조/흐름 질문 | **find-in-code** | "어디있어?", "누가 호출해?" |
| 진행 방향이 불분명 | **suggest-next** | 구체적 파일/함수 언급 없이 "기능 추가해줘" |
| 코드 구조 점검 후 구현 필요 | **codebase-health** → **propose-fix** | 진단 먼저, 구현은 이후 |
